# TiyCode 产品需求文档（PRD）

## 1. 产品概述

### 1.1 产品名称

TiyCode

### 1.2 产品定位

TiyCode 是一款开源、自由、便利的跨平台 Vibe-Coding Agent，帮助用户在本地工作区上下文中，以“线程式协作”的方式完成代码理解、方案设计、任务执行、工具调用、配置管理与扩展接入。

### 1.3 产品价值

- 将 AI 对话从“泛聊天”升级为“绑定本地项目的工作线程”。
- 将项目、线程、Git、终端、模型配置、权限策略和扩展生态放入同一桌面工作台。
- 让用户可以在一个持续上下文中完成从提问、执行到验证的完整开发闭环。
- 为后续接入真实 Agent Runtime、真实终端、真实 Git 工作流和自动化任务提供统一壳层。

### 1.4 目标用户

- 日常使用 AI 辅助编程的开发者。
- 需要在多个本地代码库之间切换的独立开发者或小团队成员。
- 需要可控模型配置、权限策略和扩展能力的高级用户。
- 需要将桌面端 AI 工具从“单次问答”升级为“工作台式协作”的团队。

### 1.5 当前阶段定义

当前产品处于 Alpha 原型向正式产品收敛阶段。页面已完整表达核心产品形态，但部分能力仍为演示态或本地模拟态，本 PRD 采用“目标能力 + 当前页面表达方式”的口径描述。

### 1.6 本期目标

- 建立完整的桌面 AI 工作台产品框架。
- 明确工作区、线程、配置中心、扩展中心之间的关系。
- 验证线程式任务协作体验是否成立。
- 为真实执行能力接入预留稳定的信息架构与交互骨架。

### 1.7 非目标

- 本期不定义团队协作、云同步、多用户权限体系。
- 本期不定义完整商业化计费逻辑。
- 本期不引入与当前页面无关的新业务线，如知识库平台、任务看板、团队管理后台等。

## 2. 产品对象定义

### 2.1 Workspace

Workspace 是用户在 Tiy Agent 中绑定的本地工作目录，是线程、项目树、终端和工具执行的基础上下文。

- 关键字段：名称、路径、是否默认工作区、是否 Git 仓库、是否自动工作树模式。
- 核心职责：为新线程提供上下文入口，为项目树和工具执行提供路径边界。

### 2.2 Thread

Thread 是围绕单个任务持续展开的对话与执行单元。

- 关键字段：标题、所属 Workspace、状态、最近活跃时间、是否当前激活。
- 核心职责：承载任务对话、计划、推理、工具执行、后续追问和结果沉淀。

### 2.3 Agent Profile

Agent Profile 是用户定义的代理风格配置。

- 关键字段：名称、自定义指令、响应风格、响应语言、主模型/辅助模型/轻量模型映射。
- 核心职责：决定线程内的默认回答风格与模型偏好。

### 2.4 Provider

Provider 是模型提供方配置实体。

- 关键字段：供应商名称、Base URL、API Key、协议类型、启用状态、模型列表、自定义请求头。
- 核心职责：为工作台提供可切换、可治理的模型能力来源。

### 2.5 Command

Command 是可在对话输入区通过 `/` 触发的预置命令模板。

- 关键字段：命令名、路径、参数提示、描述。
- 核心职责：将高频复杂操作标准化为可复用的提示入口。

### 2.6 Policy

Policy 是工具执行和环境访问的统一权限策略对象。

- 关键字段：审批模式、沙箱模式、网络策略、允许规则、拒绝规则、可写目录列表。
- 核心职责：控制 Agent 对本地环境和网络资源的访问边界。

### 2.7 Marketplace Item

Marketplace Item 是扩展生态中的可安装能力单元。

- 关键字段：类别、来源、版本、推荐状态、安装状态、启用状态。
- 核心职责：承载技能、MCP、插件和自动化任务等扩展能力。

### 2.8 Automation

Automation 是 Marketplace 中的计划任务型能力。

- 关键字段：名称、类别、安装状态、启用状态、摘要描述。
- 核心职责：表达未来可按周期运行的自动化工作流能力。
- 当前口径：视为 Marketplace 的一类扩展，不单独定义新的调度系统规则。

## 3. 信息架构与全局工作台结构

### 3.1 全局结构

产品采用桌面工作台布局，核心由五个区域组成：

- 顶部 Top Bar：承载用户入口、全局设置快捷切换、窗口控制、面板开关。
- 左侧 Sidebar：承载新线程入口、Marketplace 入口、Workspace 与 Thread 列表。
- 中央 Main Content：承载新线程态或当前线程态的核心内容。
- 右侧 Drawer：承载项目树与 Git/SCM 等辅助上下文面板。
- 底部 Terminal：承载终端区域，未来用于真实线程级执行反馈。

### 3.2 全局浮层

- Settings Center：全屏设置中心，管理账号、通用偏好、Provider、Command、Permissions、Workspace、About。
- Marketplace：全屏扩展中心，管理 Skills、MCP、Plugins、Automations。

### 3.3 导航原则

- 线程是主任务入口，设置与扩展通过全屏浮层进入。
- 工作区与线程关系固定：线程属于某个工作区，切换线程即切换主要任务上下文。
- 右侧 Drawer 与底部 Terminal 是辅助面板，不替代中央线程内容。
- 全局采用固定视口桌面模型，不使用页面级滚动。

## 4. 核心业务流

### 4.1 选择工作区并发起新线程

1. 用户进入工作台默认看到 New Thread 状态。
2. 用户先选择一个最近项目或新文件夹作为 Workspace。
3. 用户在输入区输入任务请求并提交。
4. 系统以当前 Workspace 为上下文创建新线程，写入左侧对应工作区下。
5. 系统自动切换到 Active Thread 视图，并展示线程标题与运行状态。

成功标准：

- 用户必须先明确工作区，再进入任务线程。
- 新线程必须归属某个工作区。
- 新线程创建后，工作区列表和最近项目列表都能反映最新活跃状态。

### 4.2 在线程内完成任务对话、计划、推理、工具调用与追问

1. 用户进入某一线程查看任务内容。
2. 系统展示围绕该任务的计划、队列、推理、来源、工具执行与确认流。
3. 用户可继续发送追问，并可切换 Agent Profile。
4. 系统基于当前 Profile 的风格和模型映射生成后续响应。
5. 系统通过状态和结构化组件持续反馈任务进度。

成功标准：

- 线程内容不是单纯聊天记录，而是任务执行工作流。
- Profile 切换应影响后续交互，而不是改变历史消息。
- 用户可清晰区分请求、推理、工具执行、结果来源和待确认动作。

### 4.3 在项目树、Git 抽屉和终端之间协同完成开发任务

1. 用户在右侧打开项目树查看当前工作区文件结构。
2. 用户可按文件名过滤、查看目录结构，并选择外部应用打开工作区。
3. 用户切换至 Git 面板查看变更、暂存状态、Diff 预览和历史。
4. 用户通过底部终端面板查看执行反馈或未来真实命令输出。
5. 用户在中央线程中继续与 Agent 协作，形成“对话 + 项目 + Git + 终端”的闭环。

成功标准：

- 右侧面板提供辅助上下文，不打断主线程。
- Git 和终端应服务于线程任务，而不是独立孤岛。
- 用户可以在同一工作台中完成查看、判断和执行。

### 4.4 在 Settings 中完成模型、Profile、命令、权限和工作区配置

1. 用户通过顶部用户入口进入设置中心。
2. 用户管理主题、语言、启动行为等通用偏好。
3. 用户配置 Provider 与模型清单。
4. 用户创建或编辑 Agent Profile。
5. 用户维护命令模板、审批模式、沙箱与网络策略、工作区列表。

成功标准：

- 设置中心既服务普通用户偏好，也服务高级开发者控制需求。
- 模型、权限、工作区三类配置必须能形成闭环。
- 所有设置均以本地持久化为基础。

### 4.5 在 Marketplace 中安装、启用、停用生态能力

1. 用户从左侧进入 Marketplace。
2. 用户在 Skills、MCP、Plugins、Automations 间切换。
3. 用户可搜索、查看已安装与推荐项、查看详情。
4. 用户可执行安装、启用、停用、卸载等操作。
5. 系统以安装状态与启用状态持续反馈扩展可用性。

成功标准：

- Marketplace 不只是展示目录，而是扩展能力管理中心。
- 已安装与推荐必须明确分层。
- 安装态与启用态要区分，不混为同一概念。

## 5. 模块级需求

### 5.1 Workbench Shell

目标：提供稳定的桌面型 AI 工作台骨架。

- 必须包含顶部栏、左侧栏、中央主内容、右侧抽屉和底部终端。
- 必须支持侧边栏、抽屉、终端的展开/收起。
- 必须保证固定视口，滚动仅发生在局部容器内。
- 必须支持叠加全屏浮层而不破坏主工作台结构。

### 5.2 New Thread / Workspace Binding

目标：确保每个新任务都绑定明确的本地上下文。

- 新线程态必须要求先选择 Workspace。
- 选择器必须支持最近项目与“选择新文件夹”两类入口。
- 选择项必须展示项目名、路径摘要、最近活跃时间。
- 未绑定工作区时，不应进入正式线程协作态。

### 5.3 Active Thread Experience

目标：将线程构建为“任务协作流”而非普通聊天流。

- 线程头部必须展示标题与运行状态。
- 线程内容需要支持计划、推理、工具执行、确认、来源和后续提问等结构化能力。
- 输入区必须支持继续提问，并支持切换 Agent Profile。
- 当前页面中的 AI Elements 流程为目标形态表达，属于原型态任务叙事实现。

### 5.4 Project Drawer

目标：为当前 Workspace 提供快速结构浏览与外部打开能力。

- 必须展示当前工作区名称。
- 必须支持文件过滤。
- 必须展示项目树层级与文件类型图标。
- 必须支持“用外部应用打开当前工作区”的能力。
- 当前“打开到外部应用”已具备产品表达，但真实兼容性仍属于能力接入范围。

### 5.5 Git & Diff Drawer

目标：让用户在工作台内完成基础源码变更判断。

- 必须展示变更文件列表、文件状态、暂存状态和变更摘要。
- 必须支持单文件 Diff 预览。
- 必须支持全部暂存、全部取消暂存和填写 Commit Message。
- 必须展示提交历史和远端操作入口。
- 当前提交、拉取、推送、刷新等网络动作属于原型态交互，真实 Git 执行链路尚未接入。

### 5.6 Terminal Panel

目标：承载线程执行反馈与未来真实命令行环境。

- 必须作为底部独立面板存在，并支持收起/展开与高度调整。
- 必须与线程型工作台并存，不替代线程内容。
- 当前终端以占位输出表达产品目标能力，即未来承载真实 shell/PTY 会话。
- 后续应支持与线程上下文绑定，而不是仅作为全局静态日志面板。

### 5.7 Top Bar & Account

目标：承载轻量全局控制与账号入口。

- 必须支持主题和语言快速切换。
- 必须支持打开设置、检查更新、登录/登出和窗口控制。
- Mac 与 Windows 需适配不同桌面壳层习惯。
- 当前登录态和更新检查为本地模拟态，表达的是正式产品中的账号与版本入口。

### 5.8 Settings Center

目标：集中管理用户偏好、Agent 能力和环境策略。

账号 Account：

- 管理本地会话身份、计划与账号入口。
- 当前以本地 session 模拟展示产品形态。

通用 General：

- 管理主题、语言、启动项、默认代理行为等通用偏好。

Providers：

- 管理模型供应商、协议、API Key、模型启用状态与能力项。
- 当前已表达多 Provider、多模型配置结构。

Commands：

- 管理可复用命令模板，支撑 `/` 命令式交互。

Permissions：

- 管理审批模式、沙箱模式、网络策略、允许/拒绝规则和可写目录。
- 是产品差异化能力之一，体现开发者工具的可控性。

Workspace：

- 管理工作区列表、默认工作区与 Git 属性。
- 应与新线程入口保持一致的数据口径。

About：

- 展示版本信息、系统信息、跳转入口与更新状态。

### 5.9 Marketplace

目标：集中接入和管理 Tiy Agent 的扩展生态。

- 必须包含 Skills、MCP、Plugins、Automations 四类能力。
- 必须支持搜索、查看详情、安装、启用、停用、卸载。
- 必须区分已安装与推荐。
- 详情面板必须承载完整描述、来源、版本和二级操作。
- 当前 Marketplace 状态基于本地持久化管理，表达的是正式扩展生态产品能力。

## 6. 非功能与体验要求

### 6.1 桌面优先

- 产品以桌面工作台为主，不采用移动端页面范式。
- 信息密度应高于通用聊天产品，但不能造成拥挤感。

### 6.2 固定视口与局部滚动

- 应用需保持固定视口。
- 不允许页面级整体滚动。
- 侧边栏、线程区、设置页、Marketplace、Git 列表等均采用局部滚动。

### 6.3 多面板稳定性

- 侧边栏、右侧抽屉、终端面板的开合不能导致页面结构跳变。
- 全屏浮层出现时，主工作台结构仍应保持稳定。

### 6.4 本地持久化

- 主题、语言、设置、Marketplace 状态、用户会话等需支持本地持久化。
- 本期默认以本地存储为基础，不依赖云同步。

### 6.5 权限与安全

- 产品必须支持审批策略、沙箱策略、网络访问控制和可写目录控制。
- 该能力是面向开发者桌面 Agent 的关键基础设施，不可弱化为普通偏好设置。

### 6.6 跨平台壳层

- 产品运行于 Tauri 桌面壳层。
- 应兼顾 macOS 与 Windows 的标题栏、窗口控制与系统行为差异。

### 6.7 主题与语言

- 必须支持 System / Light / Dark 三种主题偏好。
- 必须支持中英文语言切换。
- 主题与语言切换应为全局即时生效。

### 6.8 可访问性与快捷操作

- 关键区域需支持键盘交互。
- `Esc` 应可关闭浮层或退出局部预览态。
- 高密度界面中，选中态、激活态、禁用态和状态色必须清晰可辨。

## 7. 当前实现状态与后续演进

### 7.1 已清晰表达的正式产品能力

- 桌面工作台整体信息架构。
- 工作区与线程的核心关系。
- 设置中心的能力模型。
- Marketplace 的扩展分类与管理模式。
- Git、项目树、终端作为辅助执行面板的产品位置。
- Agent Profile、Provider、Policy 等关键对象边界。

### 7.2 当前仍属原型态或本地模拟态的能力

- 登录态与账号体系。
- 线程内 AI 执行与真实工具调用链路。
- 底部终端的真实 PTY 会话。
- Git 的真实提交、拉取、推送和远端同步。
- 更新检查的真实版本服务。
- Marketplace 的真实远端目录、安装与运行机制。
- 工作区在外部应用中的打开兼容性闭环。

### 7.3 建议的后续演进顺序

第一阶段：

- 接入真实线程运行时与消息执行状态。
- 打通 Settings 中 Profile、Provider、Policy 与真实运行逻辑。

第二阶段：

- 将底部终端升级为真实线程级终端会话。
- 将 Git 面板升级为真实仓库状态与操作面板。

第三阶段：

- 接入真实 Marketplace 后端与扩展安装机制。
- 补齐账号、更新检查、自动化任务的真实产品闭环。

## 8. 本期验收口径

- 用户能以“选择工作区 -> 发起线程 -> 在线程中协作 -> 查看项目/Git/终端 -> 调整设置 -> 管理扩展”的路径理解整个产品闭环。
- 所有一级产品面都能在统一工作台信息架构下成立，不互相冲突。
- 设置中心、Marketplace、线程主视图三者职责边界明确。
- 原型态能力在文档中均被明确标记，不与已实现正式能力混淆。
- 文档可同时被产品、设计和研发团队作为阶段性统一口径使用。
